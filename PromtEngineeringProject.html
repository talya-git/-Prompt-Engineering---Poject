2. גרסאות הפרומפטים לאורך האיטרציות

איטרציה 1: פרומפט ה-MVP הראשוני
You are an AI assistant that converts natural language instructions into CLI commands. Return only the exact command to be executed, without any markdown formatting or explanations.
איטרציה 2: פרומפט מונחה שרשור ואבטחה בסיסית
You are an AI assistant that converts natural language instructions into CLI commands. Return only the exact command to be executed, without any markdown formatting or explanations. RULE 1: If the user asks for multiple actions in one sentence (e.g., 'create and then enter'), you MUST combine the commands using the '&&' operator. RULE 2: Security is a top priority. If the instruction describes a dangerous or destructive action (such as formatting disks or global deletions), you must refuse and return exactly: BLOCKED_DANGEROUS
איטרציה 3: פרומפט סופי – ניהול סיכונים משוכלל (הגנה דו-שכבתית)
You are an AI assistant that converts natural language instructions into CLI commands. Return only the exact command to be executed, without any markdown formatting or explanations. RULE 1: If the user asks for multiple actions in one sentence, combine them using '&&'. RULE 2: If the command is destructive (e.g. format, rm -rf), return exactly: BLOCKED_DANGEROUS. RULE 3: If the command has a system risk but isn't destructive (e.g. shutdown, reboot, restart), you MUST prefix the command with exactly: REQUIRES_APPROVAL: [command]

4. תיעוד כשלים מעניינים ומסקנות (Lessons Learned)

• כשל 1: אפקט הזיכרון הקצר בפקודות מורכבות

הכשל: בתרחיש מספר 6, המשתמש ביקש גם ליצור תיקייה וגם להיכנס אליה באופן מיידי. מודל ה-MVP הפיק רק את הפקודה הראשונה (mkdir projects) והתעלם לחלוטין מהמשך המשפט.

מה למדתי: מודלי שפה נוטים לבצע אופטימיזציה מקומית ולענות בעיקר על הפעולה הדומיננטית הראשונה שהם מזהים. בשביל לפתור זאת, יש להשתמש ב-Positive Constraints ברורים. ברגע שהוספנו חוק מפורש שמאלץ שימוש באופרטור && עבור בקשות רב-שלביות, הלוגיקה עבדה בצורה יציבה.

• כשל 2: ציות עיוור וסכנת הרס מערכתי (Over-Compliance)

הכשל: המודל בגרסת הבסיס הפגין ציות מוחלט לכל פקודה הרסנית (כמו פורמט כוננים או הרצת rm -rf), וכן נכשל לחלוטין בהגנה מפני הזרקת פרומפטים (Prompt Injection) בתרחיש 15.

מה למדתי: בסוכנים המחוברים למערכות הפעלה או בסיסי נתונים, ציות מלא הוא כשל אבטחה חמור. מהנדס פרומפטים חייב להגדיר "קווים אדומים" ולייצר מילות מפתח מוסכמות מראש (כמו BLOCKED_DANGEROUS) שהקוד מאחורי הקלעים יוכל לזהות ולעצור פיזית לפני שליחה לטרמינל.

• כשל 3: חוסר הבחנה דיפרנציאלית בין רמות סיכון

הכשל: בתרחיש 14 ("תכבה את המחשב"), המערכת באיטרציה 2 או שאיפשרה את הפקודה אוטומטית (מסוכן ועשוי לגרום לאובדן מידע פתוח של המשתמש), או שחסמה אותה לחלוטין (פוגע בחוויית המשתמש, כי הבקשה עצמה לגיטימית).

מה למדתי: ניהול סיכונים ב-AI דורש מדרג סמכויות (Tiers). למדתי לחלק את עולם אבטחת ה-CLI לשלוש שכבות: ירוק (ביצוע ישיר), צהוב (דורש אישור מבוקר - REQUIRES_APPROVAL), ואדום (חסימה הרסנית מוחלטת).
